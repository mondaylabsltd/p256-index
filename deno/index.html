WebAuthn P256 Public Key Index Service

WebAuthn P256 Public Key Index Service. Data is stored in a smart contract (V2) on Gnosis Chain; this service acts as a read/write proxy, exposing a REST API.

Contract source: webauthnp256-publickey-index-contracts

Contract Address (Gnosis Chain)
WebAuthnP256PublicKeyIndex 0xdd93420BD49baaBdFF4A363DdD300622Ae87E9c3
WebAuthnP256BatchHelper 0xc7B0db5d4974abA3EA25780f40Bf369CC013a16E

Public endpoint: https://webauthnp256-publickey-index.biubiu.tools

  • Runtime: Deno (self-hosted) or Cloudflare Workers (edge deployment), two independent deployments
  • Data source: Gnosis on-chain contract (read/write via viem, RPC polling + automatic failover)
  • Write path: async queue (Deno: SQLite / CF: D1), background batch commit-reveal on-chain
  • Batch on-chain: via BatchHelper contract, single tx handles multiple commit/createRecord calls
  • Dual wallet: commit and createRecord use separate EOAs, nonces never interfere
  • Default port: 11256 (Deno) / N/A (CF Workers)
  • CORS: allows all origins

API Reference

Base URL: https://webauthnp256-publickey-index.biubiu.tools (or your self-hosted address)

GET /api/challenge

Get a random challenge (backward compatible).

Response (200):

{
  "challenge": "a1b2c3d4..."
}

POST /api/create

Create a public key record. The request returns 202 immediately; the on-chain commit-reveal flow runs asynchronously in the background.

Request body (JSON):

Field Type Required Description
rpId string Yes Site domain (e.g. example.com)
credentialId string Yes Credential ID
publicKey string Yes P256 public key (hex, uncompressed with 04 prefix, 65 bytes)
name string Yes Display name of the passkey
walletRef string No Wallet identifier (bytes32 hex), defaults to a deterministic Safe address derived from publicKey
initialCredentialId string No Initial credential ID, defaults to credentialId (points to root credential during key rotation)
metadata string No Additional metadata (hex), defaults to abi.encode("VelaWalletV1", publicKey)

Request example:

{
  "rpId": "example.com",
  "credentialId": "abc123",
  "publicKey": "04a1b2c3...",
  "name": "My MacBook"
}

Response (202 - Enqueued):

{
  "id": "uuid",
  "status": "pending"
}

Response (201 - Already exists, idempotent):

{
  "rpId": "example.com",
  "credentialId": "abc123",
  "walletRef": "0x000...abc",
  "publicKey": "04a1b2c3...",
  "name": "My MacBook",
  "status": "done"
}

Error responses:

  • 400 - Missing parameters
  • 429 - Rate limiting exceeded (5 requests per IP per minute)

GET /api/create/:id

Query the status of an async creation task.

Response - Completed (200, status=done):

{
  "id": "uuid",
  "status": "done",
  "rpId": "example.com",
  "credentialId": "abc123",
  "walletRef": "0x000...abc",
  "publicKey": "04a1b2c3...",
  "name": "My MacBook",
  "txHash": "0x...",
  "createdAt": 1711000000000
}

Response - In progress (200, status≠done):

{
  "id": "uuid",
  "status": "pending | committed",
  "rpId": "example.com",
  "publicKey": "04a1b2c3...",
  "name": "My MacBook",
  "createdAt": 1711000000000
}

While in progress, credentialId and walletRef are not returned to prevent front-running during the commit-reveal phase. On failure, an error field is included.

Status machine: pending → committed → done. Automatic retry up to 3 times on failure.

GET /api/query

Query public key records. Two query methods are supported:

Method 1: By rpId + credentialId

GET /api/query?rpId=example.com&credentialId=abc123

Method 2: By walletRef

GET /api/query?walletRef=0x000...abc

Success response (200):

{
  "rpId": "example.com",
  "credentialId": "abc123",
  "walletRef": "0x000...abc",
  "publicKey": "04a1b2c3...",
  "name": "My MacBook",
  "initialCredentialId": "abc123",
  "metadata": "0000...00",
  "createdAt": 1711000000000
}

If not found on-chain but present in the queue (being submitted on-chain), partial data from the queue is returned with a _queue field (credentialId/walletRef/initialCredentialId are hidden to prevent front-running):

{
  "rpId": "example.com",
  "publicKey": "04a1b2c3...",
  "name": "My MacBook",
  "metadata": "0000...00",
  "createdAt": 1711000000000,
  "_queue": { "id": "uuid", "status": "committing" }
}

Error responses:

  • 400 - Missing parameters (requires rpId+credentialId or walletRef)
  • 404 - Not found

Full Call Flow Example

1. POST /api/create                           → 202 { id, status: "pending" }
   Body: { rpId, credentialId, publicKey, name }
2. GET  /api/create/:id                       → { status: "done", txHash: "0x..." }
3. GET  /api/query?rpId=...&credentialId=...  → { publicKey, walletRef, ... }
   or GET /api/query?walletRef=0x...          → same as above

Privacy & Compliance

  • IP addresses are stored after SHA-256 hashing; the original IP cannot be recovered, GDPR compliant
  • No user identity information is collected; only public keys and site domains (public data) are stored
  • All data is immutable once on-chain (blockchain property); users should be aware of this

Caching Strategy

Dual-layer caching:

  • Server-side in-memory cache: 5-minute TTL, 100 MB cap, oldest entries evicted when full
  • CDN cache: Cache-Control: public, max-age=3600 (1 hour)

Rules:

  • Only 200 responses are cached; 404 responses are not cached

Local Development

# Deno
deno task dev          # Hot-reload dev (auto-loads .env)
deno task test         # Run tests (69 tests)

# Cloudflare Worker
npm install            # Install dependencies
npm test               # Run tests (vitest + miniflare)
npm run dev            # Local wrangler dev

Deployment (Deno - Self-hosted Server)

Deploy to a server via deno task deploy, interactively selecting the target.

deno task deploy              # Deploy: select target → upload → health check
deno task deploy status       # Check remote service status
deno task deploy rollback     # Roll back to previous version

First deployment automatically: installs Deno → creates user/directories → prompts for .env configuration → installs systemd service.

Deployment (Cloudflare Workers)

npx wrangler secret put PRIVATE_KEY            # Set Secrets
npx wrangler secret put TELEGRAM_BOT_TOKEN
npx wrangler secret put TELEGRAM_CHAT_ID
npm run deploy                                 # Deploy

Queue processing is handled via Durable Object + Alarm (~10s interval), with a Cron Trigger (every minute) as a backup to ensure the DO alarm is running.
